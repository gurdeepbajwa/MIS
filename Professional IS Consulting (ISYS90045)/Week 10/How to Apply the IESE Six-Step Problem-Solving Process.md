# How to Apply the IESE Six-Step Problem-Solving Process

Course: [[Professional IS Consulting (ISYS90045)/Course Overview|Professional IS Consulting (ISYS90045)]]
Week: 10
Source: [[Professional IS Consulting (ISYS90045)/Week 10/Lecture|Week 10 Lecture]]
Diataxis type: How-to guide

## Purpose

Use the IESE six-step process to move from an unstructured consulting problem to a defensible decision and action plan.

## When to Use It

Use this process when a client problem:

- is ambiguous or unstructured;
- has several possible options;
- requires trade-offs across cost, risk, people, technology, timing, or business value;
- needs a recommendation the client can act on.

## Steps

1. Define the problem.
   - Clarify the real issue, decision owner, scope, symptoms, possible causes, and constraints.
   - Frame the problem so the client or decision maker can actually act on it.
2. Select criteria.
   - Decide how options will be judged before favouring an answer.
   - Include tangible results, risk, and stakeholder effects.
3. Develop alternatives.
   - Generate meaningful options that respond to the defined problem.
   - Make sure the alternatives match the problem definition. If the problem changes, the alternatives may need to change.
4. Analyse alternatives.
   - Compare options against the criteria using evidence and judgement.
   - Use quantitative analysis where possible, but make subjective judgements explicit rather than hiding them.
5. Decide.
   - Recommend the option that best fits the client context, trade-offs, and constraints.
   - Do not pretend the chosen option has no weaknesses. Explain why its trade-offs are acceptable.
6. Create an action plan.
   - Translate the decision into steps, owners, sequencing, resources, risks, and follow-up.
   - Use the action plan to maximise the selected option's strengths and mitigate its downsides.

## Criteria Examples

| Criterion | Why It Matters |
| --- | --- |
| Business value | Tests whether the option addresses the client objective. |
| Cost | Tests affordability and resource impact. |
| Implementation risk | Tests whether the option can be delivered safely. |
| Stakeholder impact | Tests adoption, resistance, workload, and trust effects. |
| Time to value | Tests how quickly benefits can be realised. |
| Capability fit | Tests whether the client can maintain the solution. |

## Common Mistakes

| Mistake | Better Practice |
| --- | --- |
| Starting with a preferred option | Define the problem and criteria first. |
| Treating criteria as an afterthought | Use criteria to make trade-offs explicit. |
| Generating only one alternative | Compare meaningful options, including doing less or staging change. |
| Analysing forever | Analyse deeply enough to support a decision, not to remove all uncertainty. |
| Ending at the recommendation | Include an action plan with owners, sequence, resources, and follow-up. |
| Framing the problem outside the client's control | Reframe it into a decision or action the client can influence. |
| Ignoring human or political effects | Treat stakeholder impact as a real decision criterion. |

## Exam Use

Use this structure for consulting scenario questions that ask what the consultant should recommend or how the consultant should solve an unstructured problem.

A strong answer should show:

- problem definition before solution;
- criteria before option selection;
- alternatives and trade-offs;
- recommendation;
- practical action plan.

## Retrieval Prompts

1. Why should criteria be selected before alternatives are favoured?
2. What makes a consulting problem unstructured?
3. Why is the action plan part of the solution?
4. How does client capability affect the quality of a recommendation?
